Tests unitaires: tester un projet C (embarqué) avec Ceedling

Mar 15, 2026
#Tests unitaires#Code#C

Tester son code

Aaaah, les tests… Probablement l’un des sujets qui divise le plus dans un bureau de dev'.

« C’est inutile, ça prend trop de temps, le client testera. » « Il n’y a pas de tests ? On ne déploie rien. »

Entre ceux qui ne voient que par le TDD et les afficionados du printf("ça marche\n");, il y a peut-être un juste milieu.

Personnellement, je me range du côté de ceux qui pensent que la présence de tests, quels qu'ils soient, fait partie des critères de validation d'un développement — même en C. Surtout en C.

Mais alors, comment tester son code ? Tests de bout en bout (E2E), d'intégration, unitaires, jéjé qui clique toute la journée et qui regarde si ça pète ? Même si la dernière était juste une petite boutade, en réalité, toutes les méthodes ont leur intêret et se complètent. À mon avis, le plus important c'est de se lancer et de commencer avec quelque chose, puis d'améliorer sa stratégie au fur et à mesure.

Un seul test écrit, c’est déjà une fonctionnalité couverte.

Dans cet article, je vais aborder le sujet des tests unitaires en C avec Ceedling. Pourquoi Ceedling ? Car c'est un outil complet, pensé pour le C, largement utilisé en embarqué, simplifiant énormément la mise en place d’une vraie stratégie de tests — sans transformer votre projet en usine à gaz.

L’objectif ici n’est pas de devenir expert en tests en 15 minutes, mais de comprendre comment intégrer proprement des tests unitaires dans un projet C. Je ne vais pas non plus vous faire un cours magistral sur les tests unitaires — il existe déjà des centaines d’articles très bien faits sur le sujet. Néanmoins, je pense qu’un bref rappel est utile pour bien comprendre les enjeux, surtout dans un environnement embarqué (ou proche).

Les risques d'un développement "à l'arrache"

La liste est looooooongue.. En voici un petit aperçu, non exhaustif:

Les 3 mois de développement annoncés qui s'étalent sur l'année

On pense gagner du temps en ne testant pas. Dans la réalité :

  • Le développeur passe son temps à corriger, car la fonctionnalité ne fonctionne réellement qu'après quelques itérations.
  • La recherche de bugs devient interminable: on ajoute des printf, on recompile, on reflashe, on recommence.
  • La validation manuelle est fastidieuse, répétitive et tout les cas ne sont jamais couverts.
  • Le refactoring est repoussé depuis des mois, parce que "ça marche comme ça" et que personne n’a envie de découvrir ce que ça casse.

Chaque bug coûte plus cher que s’il avait été détecté immédiatement.

Les régressions et effets de bord

Le module écrit il y a 3 ans, 1200 lignes, 14 variables globales, 8 dépendances croisées.. Bref personne n’ose y toucher. Tu corriges un bug ? Tu en créé deux nouveaux.

Par le passé, un ami comparait souvent les correctifs sur des vieux softs sans tests ni documentation à une pelote de laine. Tu penses démêler en tirant doucement sur un fil, au final tu ne fais que créer un nouveau noeud dans une pelote déjà emmêlée.

Pas de tests → pas de garde-fous. Chaque modification devient une expérience en production. Le client teste, et se plaint.

Code peu (ou pas) fiable

Moins on teste, moins on maîtrise. Les bugs n’apparaissent pas toujours immédiatement, et peuvent provenir de n'importe où: Une valeur limite, un overflow/underflow, une condition rare, un reboot inattendu, un rayon cosmique (non pour celui là je rigole, même si c'est peut être déjà arrivé)

On peut penser au cas du Zune 30, bloqué le 31 décembre 2008 à cause d’un bug lié au calcul d’année bissextile - un cas limite, non anticipé, impact mondial.

Dormir tranquille grâce aux tests unitaires

« Mais alors, qu'est ce que je peux faire pour éviter ça ? »

Les tests unitaires constituent une véritable stratégie de gestion du risque, de documentation et d’amélioration continue. En investissant dans une bonne couverture de tests dès le début du projet, on gagne en stabilité, en rapidité de développement et en confiance collective, tout en réduisant les coûts de maintenance à long terme.

Un code réellement conforme aux attentes

Avec la mise en place des tests unitaires, lorsqu'une fonctionnalité est demandée et qu'un cahier des charges est rédigé, le développeur peut en déduire des cas de tests. Le code pourra alors y être soumis, et le développement ne sera validé que lorsque l'entièreté des tests seront passés.

Le développeur est en mesure d'au moins estimer clairement à quel point son développement est conforme et couvre les attentes du cahier des charges. Avant: « Oui normalement ça marche. » — après: « Je t'assure que dans ce cadre là, ça fonctionne! ».

Une "auto-documentation"

Les tests unitaires sont une véritable documentation exécutable pour le code qu'ils couvrent. Entre le listing exhaustif des fonctionnalités, la démonstration de l'usage des fonctions de l'API et la mise en lumière des cas aux limites, il ne reste plus qu'à y insérer un schéma et une petite introduction pour disposer d'une base de documentation solide.

C'est aussi une méthode très pérenne pour la documentation. Etant donné que les tests suivent l'évolution du code, la documentation est toujours à jour! ;)

Une architecture mieux pensée

Je ne dis pas que sans tests unitaires, il est impossible de concevoir une architecture correcte.. L'inverse n'est pas toujours vrai non plus. Mais la mise en place des TUs force le développeur à réfléchir, et cette réflexion permet d'éviter une dette technique potentiellement fatale pour un projet.

Adieu les 36 dépendances imbriquées, les fonctionnements bancales et les corner case cachés; place aux workflow clairs et aux API proprement structurées!

Le filet de sécurité dont rêvent tous les développeurs

Mise en contexte: Vous ajoutez une fonctionnalité sur un code déjà existant. Vous touchez un peu partout, parfois même à des endroits dont vous n'avez pas une maîtrise totale. L'angoisse monte... « J'espère que je n'ai rien cassé.. » « C'est le socle principal de l'application.. ». Vous découvrez alors un dossier tests/, contenant un readme indiquant les étapes à suivre pour les exécuter.

Quelques secondes plus tard, vous lancez les tests. Les tests unitaires deviennent alors un véritable filet de sécurité : ils permettent de vérifier rapidement que vos modifications n’ont pas introduit de régressions et offrent la confiance nécessaire pour continuer à faire évoluer le code sereinement — même lorsque plusieurs développeurs travaillent dessus.

Bien entendu avant de partir, vous vous assurez d'avoir ajouté les tests nécessaires pour couvrir le code que vous venez d'introduire ;)

Disclaimer: aucune solution n'est parfaite

Eh oui, je vous mentirais en vous disant que la mise en place de tests unitaires ne venait pas avec son lot d'inconvénients. Il faut en être conscient pour pouvoir les maitriser par la suite.

Dans un premier temps, leur mise en place peut être fastidieuse, surtout sur un code déjà écrit. Néanmoins, c'est un point qui peut être rapidement effacé avec une méthodologie efficace (TDD par exemple).

Ensuite, les tests unitaires par essence, suivent le développement du code testé. Il doivent donc être maintenus et entretenus. C'est donc une tâche à prendre en compte dans le temps de développement (temps largement rattrapé par la suite).

Enfin, et c'est inhérent à toute solution de qualité logicielle, ces tests doivent être exécutés. En effet, s'ils ne sont pas lancés après chaque correction ou ajout de nouvelle fonctionnalité, autant dire qu'il ne servent à rien.

Les tests unitaires ne sont pas LA seule solution, mais leur mise en place est clairement conseillée afin de garantir une certaine qualité logicielle.

Démarrage rapide avec Ceedling

Ceedling est un outil en ligne de commande qui facilite la mise en place et l’exécution de tests unitaires dans des projets écrits en C. En somme, c'est un build system, permettant de construire un environnement de test complet pour organiser et compiler les tests, les exécuter et afficher le résultat.

Pour fonctionner, Ceedling s’appuie sur deux bibliothèques principales (développées par le même éditeur) :

  • Unity — le framework de test contenant tous les utilitaires d'assertions et de contrôle.
  • CMock — l'outil qui permet de mock automatiquement les fonctions interface afin de simuler les dépendances lors des tests.

Cet ensemble d'outil s'intègre sans encombre à côté du logiciel développé. C'est une solution non intrusive, puisque le code n'est absolument pas modifié et est compilé avec l'environnement de test par un compilateur C standard (GCC par exemple). Toute la configuration est contenue dans un fichier project.yml (sources du projet, plugins, ...)

L'exécutable de test généré par Ceedling est parfaitement standard, et peut donc être débuggué et/ou analysé avec les outils habituels (GDB, Valgrind, ...)

Installation

L'installation de Ceedling est très simple. Dans un premier temps, assurez-vous d'avoir installé Ruby. Si vous n'en n'avez jamais entendu parlé, je vous invite à parcourir l'article dédié à la présentation du language (Rassurez-vous, nul besoin de connaître Ruby pour utiliser Ceedling..). Enfin, installez la gem Ceedling.

  > gem install ceedling

Pour finir, vérifiez votre installation.

  > ceedling version

Guide d'utilisation

Pour la démonstration, je vais m'appuyer sur une petite lib que j'ai développé: Broki. Agissant comme un broker, celle-ci permet de créer des fluxs internes appelés streams. Les producers peuvent alors envoyer des données à travers ces fluxs, qui seront reçus et traités par des subscribers. Chaque flux est identifié uniquement par un id. Même la taille maximum du buffer pouvant être transmis est spécifiée lors de la création, les producers peuvent tout à fait émettre un buffer avec une taille inférieure. C'est utile avec du protobuf par exemple, qui définit un nombre max d'octets pour le payload, mais dont la taille dépend des données à sérialiser.

/**
 * @brief Function used to initialize Broki library.
 * This function must be called before every other ones (cleans up the context, ...).
 */
broki_error broki_init( void );

/**
 * @brief Function used to create a data stream.
 * @param id Id of the stream to create
 * @param payload_size Maximum payload size that will go through the stream
 * @return broki_error
 */
broki_error broki_create_stream( uint32_t id, uint32_t payload_size );

/**
 * @brief Function used to get the payload size of a data stream.
 * @param id Id of the stream
 * @param payload_size Pointer on payload size holder
 * @return broki_error
 */
broki_error broki_get_stream_payload_size( uint32_t id, uint32_t* payload_size );

/**
 * @brief Function used to send data through a stream.
 * @param id Id of the stream
 * @param payload_size Pointer on payload size holder
 * @return broki_error
 */
broki_error broki_send_in_stream( uint32_t id, void* data, uint32_t payload_size );

/**
 * @brief Function used to subscribe to a stream.
 * @param id Id of the stream
 * @param callback Callback to receive data changes
 * @return broki_error
 */
typedef void (*broki_stream_callback)( uint32_t id, void* payload, uint32_t payload_size );
broki_error broki_subscribe_stream( uint32_t id, broki_stream_callback callback );

Dans ce cas précis, les fonctionnalités à tester sont assez claires:

  • être en mesure de créer un stream
  • pouvoir y envoyer des données (producer)
  • vérifier la bonne transmission (subscriber)

Toute la configuration est stockée dans un seul fichier YAML project.yml. Cette configuration indique à Ceedling où trouver les fichiers source, de quelle manière éxecuter les tests, quels outils/plugins sont utilisés, et bien plus encore.

Pour démarrer un projet:

  > ceedling new MyProject
        create  MyProject
        create  MyProject/src
        create  MyProject/test
        create  MyProject/test/support
        create  MyProject/project.yml

  🌱 New project 'MyProject' created at ./MyProject/

  > cd MyProject

Une arborescence de base est générée, accompagnée d'un template de fichier project.yml. Vous n’êtes pas obligé de garder cette organisation pour votre projet. Pour la modifier, il suffira simplement de mettre à jour la section :paths: dans le fichier project.yml.

# This is where Ceedling should look for your source and test files.
# see documentation for the many options for specifying this.
:paths:
  :test:
    - +:test/
    - -:test/support
  :source:
    - src/
  :include:
    - src/ # In simple projects, this entry often duplicates :source
  :support:
    - test/support
  :libraries: []

Le projet est quand même un peu vide... il est possible d'utiliser Ceedling pour créer une paire de fichier .c/.h avec son fichier de test.

  > ceedling module:create[Demo]
    🚧 Loaded project configuration from working directory.
    File src/Demo.c created
    File src/Demo.h created
    File test/test_Demo.c created
    Generate Complete

Pour un exemple concret, n'hésitez pas à aller faire un tour sur une lib que j'ai développé.

Pro tips

Par défaut, Ceedling identifie les fichiers de tests grâce à leur préfixe test_.

  :test_file_prefix: test_

CMock identifie les fichiers grâce à leur préfixe mock_ pour le mocking automatique.

#include "mock_lib_driver.h"

Pour activer la gestion du coverage via Gcov, il suffit de décommenter la ligne correspondante dans la liste des plugins. De la même manière, il est possible d'activer la génération d'un fichier compile_commands.json.

:plugins:
  :load_paths: []
  :enabled:
    #- beep
    - module_generator
    - gcov <--
    #- bullseye
    #- command_hooks
    - compile_commands_json_db <--
    #- dependencies
    #- subprojects
    #- fake_function_framework

Il est possible de définir des symboles qui seront utilisés pour la compilation.

# Compilation symbols to be injected into builds
# See documentation for advanced options:
#  - Test name matchers for different symbols per test executable build
#  - Referencing symbols in multiple lists using advanced YAML
#  - Specifiying symbols used during test preprocessing
:defines:
  :test:
    - TEST # Simple list option to add symbol 'TEST' to compilation of all files in all test executables
  :release: []

  # Enable to inject name of a test as a unique compilation symbol into its respective executable build. 
  :use_test_definition: FALSE

Pour lancer les tests:

  ceedling test:all
  ceedling gcov:all # Avec le Gcovr

Un fichier de test doit représenter le test d'une seule fonctionnalité (fonction par exemple). Ainsi, le contenu de la fonction setUp doit préparer le contexte à l'utilisation de cette fonctionnalité, et la fonction tearDown doit clean le contexte après utilisation de la fonctionnalité testée.

Template fichier de test 

/*
 * @file test_*.c
 * @author *
 * @brief * library testing project
 * @version *
 *
 * @copyright Copyright (c) *
 *
 */

#include "unity.h"

/* Library includes and mocks will go there */

/* Lib constants */

/* Local variables */

/* Locals */


void setUp( void ){
    /* Will be executed before each test function */
}

void tearDown( void ){
    /* Will be executed after each test function */
}

/* Tests functions will go there */
void testShouldFail( void ){
    TEST_ASSERT_EQUAL(1, 0);
}

/* Locals */
